## Backward compatibility

- **All v0.3 channels keep working unchanged** — email, webhook (no `secret`), Telegram, Slack, ntfy.
- **Webhook signature is strictly opt-in.** Existing webhook channels without a `secret` are signed nothing, just like v0.3.0. Adding `secret: "..."` (16-256 chars) on PATCH upgrades the channel to signed.
- **Backup format is independent of API version.** A tarball from v0.4.0 restores cleanly into a v0.4.0 instance. Restoring across major versions is not supported (drizzle migration 0005_audit_log.sql and 0004 from v0.2 are required).
- **OpenAPI refactor changes no URL shape.** All 15 documented routes use the same paths and methods as before. Two new endpoints added (`/api/openapi.json`, `/api/docs`).
- **Grafana adds 5 new metrics, keeps all existing names/labels.** Existing scrapers and dashboards are unaffected.
- **`/health/live` and `/health/ready`** unchanged from v0.3.0.

## Notable design choices

- **Raw body bytes for HMAC, not re-serialization.** The receiver verifies the HMAC over the exact bytes they received. This is the only sane approach — re-serializing on both sides couples the signature to your JSON formatter's whitespace choices.
- **Shelling out to `tar` for the backup CLI.** No new dependency. The backup is a CLI, not a library — shelling out is fine and gives us the user's preferred `tar` (handles sparse files, etc.) for free. `better-sqlite3`'s `.backup()` is used for the SQLite copy so the file is consistent even if writers are active.
- **OpenAPI spec is unversioned.** Derived from the live route table at startup. When someone adds a route, the spec picks it up automatically. Trade-off: external API consumers must follow the `main` branch (no `vN` snapshotting yet). Acceptable for a v0.4 release with no SDK commitments.
- **`c.req.routePath` for the `path` metric label** (not the literal URL). `/api/domains/42` and `/api/domains/7` both label as `/api/domains/:id`. This is a hard cardinality bound — without it, a 5k-domain deployment would create 5k time series per metric.
- **Per-route `describeRoute` `responses` kept slim intentionally.** Most use `responses: { 200: { description: "OK" } }` plus the Zod-derived schema for the happy path. The spec is useful (paths, methods, schemas, security) but not exhaustive (full error responses per status). Tightening this is a v0.5 cleanup — explicitly out of scope here to keep the PRs small.
- **v0.2.0 re-tagged, not deleted-and-recreated.** A force-push to a tag is the only way to make a stale tag honest. We accept the cost because (a) v0.2.0 is 4 days old, (b) SSLert has no external consumers of the tag (no Docker Hub `latest`, no brew formula, no package manager index — verified before re-tagging), (c) the original tag was always known-stale per the v0.3.0 STATUS note.
- **Tag points at code, not docs.** `v0.4.0` tag is at `2053730` (the last code commit, PR #13). The dashboard commit lands on `main` *after* the tag — this is the same pattern v0.3.0 used (tag at `d016461`, dashboard commit `d016461` was the same commit, here they're separated by intent). GitHub's release "compare" link uses the tag, not the branch tip, so the URL is stable.
